# UI & Frontend Design Guidelines

## Core Philosophy

Every interface must look like it was designed by a human with opinions, not generated by an AI hedging its bets. The enemy is "safe." Safe color choices, safe font pairings, safe layouts, safe everything. Safe produces forgettable products.

**The test**: If you screenshot the app and show it to 10 designers, would at least 3 of them say "oh that's interesting"? If not, push harder.

## Emotional Design Framework (Required)

Design every screen across three emotional layers:

- **Visceral (first impression)**: Immediate visual pull. Distinctive typography, clear hierarchy, and tactile surfaces should make users want to interact.
- **Behavioral (in-use feel)**: The UI must feel effortless. Fast feedback, obvious next actions, and minimal friction/mis-taps.
- **Reflective (memory and meaning)**: Users should feel progress and ownership (history, saved work, milestones, completion cues).

Translate this into concrete behavior loops:

1. **One obvious next action per screen** (reduce decision fatigue).
2. **Immediate feedback after every meaningful action** (visual state change, motion cue, success/error message).
3. **Visible progress** (step indicators, completion states, history) so users feel investment and return.
4. **Low-friction input ergonomics** (touch-safe targets, sensible defaults, fewer fields).

Ethical boundary (non-negotiable): build magnetic UX, not manipulative UX. Avoid dark patterns, forced confusion, fake urgency, or hard-to-exit flows.

## Typography Rules

Typography is the single biggest lever for making an app look designed vs. generated.

### Hard Requirements

- **NEVER use**: Inter, Roboto, Arial, Helvetica, system-ui, sans-serif (as primary), Open Sans, Lato, Montserrat, Poppins. These are the "I didn't make a choice" fonts.
- **Display/heading font**: Must be distinctive and characterful. Good starting points by vibe:
  - *Refined/editorial*: Playfair Display, Cormorant Garamond, Lora, Fraunces, Libre Baskerville
  - *Modern/geometric*: Archivo, Barlow, Syne, Space Grotesk (use sparingly — it's becoming overused), Outfit, Manrope
  - *Industrial/utilitarian*: JetBrains Mono, IBM Plex Mono, Overpass, Barlow Condensed
  - *Warm/humanist*: Source Serif 4, Merriweather, Nunito, DM Sans, Libre Franklin
  - *Bold/impactful*: Bebas Neue, Oswald, Anton, Unbounded, Bricolage Grotesque
- **Body font**: Must pair well with the display font. Prioritize readability at 14-16px. Serif + sans-serif pairings almost always work.
- **Monospace font** (for data, numbers, code): JetBrains Mono, IBM Plex Mono, Fira Code, Source Code Pro. Use for financial figures, timestamps, measurements, counters.
- **Font loading**: Import via Google Fonts in index.html. Define in Tailwind config. Always set fallback stacks.
- **Every app must use a different primary font**. Do not converge on the same font across projects.

### Sizing & Hierarchy

- Establish a clear type scale. Suggested: 12 / 14 / 16 / 18 / 20 / 24 / 30 / 36 / 48 / 60px
- Headings should feel noticeably different from body text — different font, weight, and often case treatment (uppercase, small-caps for labels)
- Line heights: 1.2-1.3 for headings, 1.5-1.7 for body, 1.8+ for long-form reading (generated scripts, documents, reports)
- Letter spacing: Slightly tightened for large headings (-0.02em), slightly opened for small uppercase labels (+0.05em)

## Color System

### Hard Requirements

- **NEVER use**: The default Tailwind blue (#3b82f6) as a primary color. No purple-on-white gradients. No generic blue-gray palettes. No rainbow button gradients.
- **Commit to a palette**: Every app needs a dominant color, a contrast accent, and neutrals that aren't pure gray. Warm apps get warm grays (stone, sand). Cool apps get cool grays (slate, zinc).
- **Provide exact hex codes**: Don't say "use a warm color." Say "#e8712a for CTA, #1a2332 for text, #f5f5f0 for background."
- **Status colors must exist**: Success (green-ish), warning (amber/yellow-ish), error (red-ish), info (blue-ish). These can be tinted to match the palette.
- **Dark mode is optional but deliberate**: If the industry/persona skews toward screens in dim environments (truckers at night, dispatchers), consider dark-first. Otherwise, light is fine. Don't add dark mode just because.

### Palette Construction

Build palettes with intention:

1. **Start with the industry**: What colors exist in this world? Funeral homes → earth tones, parchment. Construction → orange, navy, concrete gray. Weddings → dusty rose, sage, cream. Finance → dark navy, gold, white.
2. **Pick a dominant**: This color appears on primary CTAs, active states, key highlights. It should be instantly associated with the brand.
3. **Pick a contrast accent**: Used sparingly for secondary actions, status indicators, hover states. Should complement, not compete with the dominant.
4. **Build neutral ramp**: 5-7 shades from near-white to near-black, tinted toward your palette's temperature. Use these for backgrounds, borders, disabled states, secondary text.
5. **Test contrast**: Every text/background combination must pass WCAG AA (4.5:1 for body text, 3:1 for large text).

### Gradients & Backgrounds

- Gradients are welcome when **subtle and purposeful**. A slight warm-to-cool gradient on a hero section adds depth. A neon rainbow gradient on a button adds nothing.
- Consider textured backgrounds: subtle noise, grain, paper texture, very faint geometric patterns. These add tactile quality.
- Glassmorphism, mesh gradients, and frosted effects work when the overall aesthetic supports them. They don't work bolted onto a flat design.

## Layout & Spacing

### Hard Requirements

- **Max content width**: Most content should sit within 640-1200px depending on density needs. Full-bleed is fine for hero sections and backgrounds.
- **Whitespace is a feature**: Generous padding and margins. Don't cram elements together. If something feels "too empty," it's probably right.
- **Grid breaks are allowed**: Not everything needs to be in a 12-column grid. Asymmetric layouts, overlapping elements, and off-grid accents create visual interest.
- **Card design must be specific**: If using cards, specify: border-radius, shadow style (or no shadow), border treatment, padding, hover behavior. "A card" is not a design direction.

### Responsive Strategy

- **Mobile-first CSS**: Write the mobile layout first, then expand with breakpoints.
- **Identify the primary device**: Every app has a primary usage context. Design for that first:
  - Truckers → phone (large touch targets, chunky inputs, minimal scrolling)
  - Contractors on-site → phone/tablet (thumb-friendly, outdoor-readable contrast)
  - Office professionals → desktop (multi-column, dense data tables, keyboard shortcuts)
  - Mixed use → design the mobile experience first, then fill the desktop space meaningfully (not just center a narrow column)
- **Touch targets**: Minimum 44x44px on mobile. Prefer 48x48px for primary actions.
- **Stacking order on mobile**: Form fields go full-width. Side-by-side panels stack vertically. Tables become cards.

## Component Design Principles

### Forms

- Inputs should feel spacious: min-height 44px, generous padding (12-16px)
- Labels above inputs, not inside (no placeholder-as-label pattern)
- Focus states must be visible and on-brand (colored border or ring, not just default blue outline)
- Group related fields visually (border, background tint, or spacing)
- Textareas should be tall enough that the user doesn't immediately need to scroll (min 4 lines)
- Placeholder text should be realistic examples, not descriptions ("e.g., Margaret Anne Sullivan" not "Enter full name")
- Validation: inline, below the field, in a subdued error color with helpful text

### Buttons

- Primary: Bold, filled, dominant color, clear label ("Generate Estimate" not "Submit")
- Secondary: Outlined or ghost, lower visual weight
- Destructive: Red-tinted, with confirmation for irreversible actions
- Disabled: Clearly muted, with cursor-not-allowed
- Loading: Show a spinner or loading text inside the button, keep button width stable (no layout shift)
- Size hierarchy: One dominant CTA per screen section. Supporting actions are visually smaller.

### Tables & Data

- Use monospace for numeric columns (amounts, measurements, counts)
- Alternating row backgrounds (very subtle) or thin dividers for scanability
- Sortable columns where relevant
- Status indicators: colored dots or pills, not just text
- Empty states: Friendly illustration or message + CTA ("No estimates yet. Create your first one.")

### Cards

- If the app uses cards (history items, library items), each card must show enough info to be useful without clicking (name, date, key metric, status)
- Hover effect: subtle elevation change or border highlight
- Click target: the entire card should be clickable, not just a small link inside it

### Loading & Error States

- **Skeleton screens** for initial page loads (not blank pages or spinners)
- **Inline loading indicators** for actions (button spinners, progress bars)
- **Toast notifications** for async results (success, error) — positioned consistently (top-right or bottom-center)
- **Error boundaries** with friendly messages and retry actions
- **Empty states** with illustrations or icons and a clear next-step CTA
- **Perceived speed matters**: During waits, show meaningful progress immediately so the product never feels frozen
- **Completion feedback**: Every successful action should end with a clear confirmation state, not silent success

## Animation & Motion

### Philosophy

Motion should serve a purpose: guide attention, provide feedback, create continuity. Never animate for decoration alone.

### Implementation

- **Page transitions**: Subtle fade-in (200-300ms) on route changes
- **Content reveal**: Stagger child elements on page load (each element 50-80ms delayed). Creates a natural "drawing" feel.
- **Hover states**: Slight scale (1.02), shadow lift, or color shift. Keep under 200ms.
- **Loading/generation**: Pulsing animation, shimmer effect, or typed-text effect for AI-generated content. The user should feel something is happening.
- **State changes**: Smooth transitions between form/results modes. Don't just swap — cross-fade or slide.
- **Number animations**: Count-up animation for financial totals, usage stats, and counters. Very satisfying.
- **Timing discipline**: Most motion should stay in roughly 150-400ms ranges so interaction feels responsive, not sluggish.
- **Avoid**: Bouncing, excessive spring physics, delays over 500ms, anything that feels like it's slowing the user down.

### Tech

- Prefer CSS transitions and @keyframes for simple animations
- Use Framer Motion (React) for complex orchestration, layout animations, and AnimatePresence (enter/exit)
- Respect `prefers-reduced-motion` media query — always provide a no-motion fallback

## Visual Details That Separate Good From Great

These are the details that make someone say "this feels premium":

- **Custom scrollbar styling** (WebKit): Thin, tinted to match palette
- **Selection color** (::selection): Tinted to match brand
- **Focus-visible only**: Use `:focus-visible` not `:focus` to avoid showing focus rings on click
- **Subtle borders**: Use 1px borders in a color that's barely visible (10-15% opacity of text color), not harsh gray lines
- **Shadow layering**: Multiple layered shadows create more realistic depth than one big shadow
- **Icon consistency**: Use one icon library (Lucide, Heroicons, or Phosphor). Don't mix libraries.
- **Favicon**: Generate and include a simple favicon that matches the brand
- **Page title**: Dynamic page titles (`useEffect` to set `document.title`)
- **Print styles**: If the app generates documents (invoices, quotes, scripts), add `@media print` styles

## What "AI Slop" Looks Like (Avoid All of This)

- Purple-to-blue gradient hero with white text
- Inter font at every weight
- Identical card grids with rounded corners and light shadows
- Generic "wave" SVG dividers between sections
- Teal/purple accent buttons on white backgrounds
- "Welcome to [App Name]" as a hero headline
- Centered everything with no visual tension
- Stock-photo-style placeholder illustrations
- Every button the same size and weight
- No clear visual hierarchy — everything at the same level of importance
- Default Tailwind shadows, default Tailwind colors, default Tailwind border-radius
- Components that look like they came from a UI kit without any customization

---

## Final Check Before Shipping

Before considering the frontend done, verify:

1. [ ] The app has a distinctive color palette (not default Tailwind)
2. [ ] The heading font is characterful (not Inter/Roboto/system)
3. [ ] There's a clear visual hierarchy on every page
4. [ ] Empty states exist and have helpful CTAs
5. [ ] Loading states use skeletons or meaningful indicators
6. [ ] Mobile layout is usable with thumb-sized touch targets
7. [ ] At least one animation creates delight (page load reveal, number count-up, generation effect)
8. [ ] The landing page has an interactive element (not just text + button)
9. [ ] Forms have realistic placeholder examples
10. [ ] The app looks like nothing else you've built this week
11. [ ] Each screen has one obvious primary next action
12. [ ] Key actions provide immediate success/failure feedback
13. [ ] Progress is visible (history, completion cues, step status)
14. [ ] Engagement patterns are user-aligned (no dark patterns, no coercive loops)

server.py
import os

from twentyfirst_sdk import AgentClient

client = AgentClient(api_key=os.environ["API_KEY_21ST"])

# Create a sandbox and thread for a persistent terminal/server session.
sandbox = client.sandboxes.create(agent="my-agent")
thread = client.threads.create(sandbox_id=sandbox.id, name="Chat 1")

# Launch the agent against that thread.
run = client.threads.run(
    agent="my-agent",
    sandbox_id=sandbox.id,
    thread_id=thread.id,
    messages=[
        {
            "role": "user",
            "parts": [
                {"type": "text", "text": "Review the latest PR comments."},
            ],
        },
    ],
)

try:
    for raw_line in run.response.iter_lines(decode_unicode=True):
        if raw_line:
            print(raw_line)
finally:
    run.response.close()
